哈希表

发表于2025-03-06|更新于2025-03-06
|字数总计:2.6k|阅读时长:8分钟

哈希表核心原理

首先要区分一下，哈希表和我们常说的Map(键值映射)不是同一个东西
在Java中，Map是一个Java接口，仅仅声明了若干个方法，并没有给出方法的具体实现，而哈希表是具体实现。

几个关键概念及原理

key是唯一的、value可以重复

哈希函数

哈希函数的作用是把任意长度的输入（key）转化成固定长度的输出（索引）。
任意 Java 对象都会有一个 int hashCode() 方法，在实现自定义的类时，如果不重写这个方法，那么它的默认返回值可以认为是该对象的内存地址。一个对象的内存地址显然是全局唯一的一个整数。

所以我们只要调用 key 的 hashCode() 方法就相当于把 key 转化成了一个整数，且这个整数是全局唯一的。

又会有一个问题了，int 值可能为负数，而数组的索引是非负整数
采用位运算

上面解决了非负数，但是 hashCode一般值很大，我们需要把它映射成 table数组的合法索引

哈希冲突

如果两个不同的 key 通过哈希函数得到了相同的索引，怎么办呢？这种情况就叫做「哈希冲突」。

两种常见的解决方法，一种是拉链法，另一种是线性探查法（也经常被叫做开放寻址法）。

扩容和负载因子

避免哈希表装太满，所以有 负载因子

负载因子是一个哈希表装满的程度的度量。一般来说，负载因子越大，说明哈希表里面存储的 key-value 对越多，哈希冲突的概率就越大，哈希表的操作性能就越差。

负载因子的计算公式也很简单，就是 size / table.length。其中 size 是哈希表里面的 key-value 对的数量，table.length 是哈希表底层数组的容量。

像 Java 的 HashMap，允许我们创建哈希表时自定义负载因子，不设置的话默认是 0.75，这个值是经验值，一般保持默认就行了。

key必须是不可变类型

只有那些不可变类型，才能作为哈希表的 key，这一点很重要。

所谓不可变类型，就是说这个对象一旦创建，它的值就不能再改变了。比如 Java 中的 String, Integer 等类型，一旦创建了这些对象，你就只能读取它的值，而不能再修改它的值了。

作为对比，Java 中的 ArrayList、LinkedList 这些对象，它们创建出来之后，可以往里面随意增删元素，所以它们是可变类型。

在Java中，Map 和 哈希表（如 HashMap）是相关但不同的概念，核心区别在于 抽象接口与具体实现 的关系。以下是两者的详细区分：

一、本质区别

  1. Map是接口，哈希表是具体实现
    • Map：是Java中定义键值对（Key-Value）操作规范的接口（如put()、get()等方法），属于顶层抽象。它仅规定行为，不涉及底层数据结构
    • 哈希表：通常指HashMap（或Hashtable），是基于哈希表数据结构的具体实现类，属于Map接口的一个子类。它通过数组+链表/红黑树存储数据，并实现Map定义的方法
  2. 其他实现类对比
    • Map的常见实现类包括：
      • HashMap：基于哈希表（数组+链表/红黑树），无序存储，查询效率高。
      • TreeMap：基于红黑树，按键的自然顺序排序，查询复杂度O(log n)。
      • LinkedHashMap：基于哈希表+双向链表，保留插入顺序。
    • 哈希表仅特指HashMap或Hashtable这类基于哈希算法的实现

二、数据结构与实现原理

特性 Map（接口） 哈希表（如HashMap）
底层结构 无具体结构，仅定义行为 数组+链表/红黑树（JDK8后支持树化）
哈希函数 不涉及 通过 hashCode() 计算键的哈希值，确定存储位置
冲突解决 不涉及 链地址法（链表挂载冲突元素，树化优化）
扩容机制 无定义 负载因子触发扩容（默认0.75），数组翻倍并重哈希
线程安全性 不涉及 HashMap 非线程安全；Hashtable 线程安全（全表锁）

三、功能与特性对比

  1. 键值对操作
    • Map接口：定义通用方法（如 put(K,V)、get(K)），所有实现类必须遵循。
    • 哈希表实现：在通用方法基础上，提供哈希表特有的优化（如快速查询、动态扩容）。
  2. 允许空值
    • HashMap：允许键和值为 null。
    • Hashtable：不允许键或值为null（抛出NullPointerException）
  3. 性能与场景
    • 哈希表：适合高频查询、无顺序要求的场景，时间复杂度接近O(1)（理想情况下）
    • 其他Map实现：如需要排序用TreeMap（O(log n)），需保留插入顺序用LinkedHashMap

四、常见混淆点解析

  1. “哈希表”术语的模糊性
    • 广义上，哈希表是一种数据结构（如数组+链表）；
    • Java中通常指HashMap或Hashtable，但需注意它们与Map接口的关系
  2. Hashtable 与 HashMap 的区别
    • Hashtable是早期线程安全实现（性能低），而HashMap非线程安全但效率更高
  3. 树化优化（JDK8+）
    • 当链表长度≥8且数组容量≥64时，链表转为红黑树，避免哈希攻击导致性能退化

五、总结与选型建议

  1. 如何区分
    • Map是抽象接口：定义键值对操作的规范，不关心底层实现。
    • 哈希表是具体实现：基于哈希算法，通过数组+链表/红黑树实现Map接口。
  2. 使用场景
    • 高频查询、无需排序：优先选择 HashMap。
    • 线程安全需求：使用 ConcurrentHashMap（而非 Hashtable）。
    • 有序性要求：选择 TreeMap（自然排序）或 LinkedHashMap（插入顺序）。

在Java中，哈希表（如HashMap）是一种结合数组高效索引与链表灵活性的数据结构，其核心原理可概括为以下要点：

一、基础结构：数组 + 链表 + 红黑树

  1. 数组（主干）
    • 哈希表的主干是一个数组，每个元素称为“桶”（Bucket）。
    • 通过哈希函数计算键（Key）的哈希值，确定键值对应存入的数组索引位置。
      例如，键"name"经过哈希计算后可能落在数组的第三个位置（即索引3）
  2. 链表（解决冲突）
    • 当多个键的哈希值映射到同一数组索引时（哈希冲突），这些键值对会以链表形式挂载到对应的数组桶中。
    • 例如，键"age"和"gender"可能都映射到索引3，此时它们会形成一条链表
  3. 红黑树（优化长链表）
    • 当链表长度超过阈值（默认8），链表会转换为红黑树（平衡二叉树），将查询时间复杂度从O(n)优化到O(log n)

二、核心操作流程

1. 存储数据（PUT）

  • 步骤：
    1. 计算哈希值：通过hashCode()方法计算键的哈希值。
    2. 确定数组索引：哈希值对数组长度取模，得到存储位置
      （如index = hash % array.length）。
    3. 处理冲突：
      • 若桶为空，直接存入键值对。
      • 若桶已有数据，遍历链表或红黑树：
        • 键相同：更新值。
        • 键不同：将新键值对添加到链表末尾或树中

2. 查询数据（GET）

  • 步骤：
    1. 计算键的哈希值，确定数组索引。
    2. 遍历对应桶的链表或红黑树，通过equals()方法匹配键。
    3. 找到匹配的键后返回对应值

三、哈希冲突的解决方式

Java的HashMap采用链地址法（拉链法）解决冲突：

  1. 链表挂载：冲突的键值对以链表形式存储在同一数组桶中。
  2. 红黑树优化：当链表长度超过阈值（8）且数组长度≥64时，链表转换为红黑树，提升查询效率
  3. 其他方法对比：
    • 开放地址法：冲突后按规则探测下一个空位（如线性探测、二次探测）。
    • 再哈希法：使用多个哈希函数重新计算。
    • 公共溢出区：冲突数据存到独立区域

四、动态扩容机制

  1. 负载因子（Load Factor）
    • 默认0.75，表示当元素数量达到数组容量的75%时触发扩容。
  2. 扩容过程：
    • 新建一个容量翻倍的数组（如从16扩容到32）。
    • 重新计算所有键值对的哈希值并迁移到新数组中，以减少冲突概率

五、通俗比喻理解

想象哈希表是一个仓库：

  1. 仓库里的柜子：数组中的每个桶像一个柜子。
  2. 柜子里的链子：如果多个物品要放在同一柜子，就用链子串起来（链表）。
  3. 链子变架子：链子太长时换成多层架子（红黑树），方便快速查找。
  4. 仓库扩建：当柜子快满时，扩建仓库并重新整理物品（扩容）

六、总结

Java的哈希表通过数组+链表+红黑树的组合，在高效索引和灵活存储之间取得平衡。其核心在于：

  1. 哈希函数快速定位桶；
  2. 链地址法解决冲突；
  3. 动态扩容和红黑树优化保证性能。
    这一设计使得哈希表在增删查改操作中均能达到接近O(1)的时间复杂度（理想情况下）
